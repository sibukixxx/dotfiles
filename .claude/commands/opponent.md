# /opponent — 対立検証による意思決定支援

Anthropic社内パターン「Opponent Processor」を実装。
2つのサブエージェントが **異なる立場** で議論し、メインエージェントが統合判断する。

## 目的

- 確証バイアスを排除する
- 非相関なコンテキストウィンドウで多角的に検討する
- 「なんとなく良さそう」ではなく、根拠ある意思決定を導く

## Instructions

### Step 1: テーマの明確化

ユーザーの引数 `$ARGUMENTS` から判断テーマを特定する。
不明確な場合はヒアリングで以下を確認:

- **何を決めるのか**: 技術選定 / アーキテクチャ / 事業判断 / 投資判断
- **制約条件**: 予算、期限、チーム規模、既存資産
- **現時点の仮説**: ユーザーが傾いている方向（あれば）

### Step 2: 2エージェントの並列起動

Agent ツールで **2つのサブエージェント** を **並列** に起動する。

#### Agent A: Advocate（推進派）

```
あなたはアプローチAの強力な支持者です。

テーマ: [判断テーマ]
あなたの立場: [アプローチA] を推奨

以下の観点で、このアプローチが最善である理由を論じてください:
1. 技術的メリット（3つ以上）
2. ビジネス上のメリット（2つ以上）
3. 想定される反論への先回り反駁
4. 成功事例や類似ケース
5. 実装ロードマップの提案

制約条件: [制約]
```

#### Agent B: Challenger（反対派）

```
あなたはアプローチAに対する厳しい批評者です。

テーマ: [判断テーマ]
あなたの立場: アプローチAに異議を唱え、[アプローチB] を推奨

以下の観点で、アプローチAの問題点とBの優位性を論じてください:
1. アプローチAの技術的リスク（3つ以上）
2. アプローチAの隠れたコスト・負債
3. アプローチBが優れている理由
4. アプローチAの失敗事例や類似ケースの教訓
5. アプローチBの実装ロードマップの提案

制約条件: [制約]
```

### Step 3: 議論の統合

2つのエージェントの出力を受け取り、以下の形式で統合判断を提示する:

```markdown
## Opponent Processor: [テーマ]

### Advocate（推進派）の主張
[Agent Aの要約: 3-5行]

### Challenger（反対派）の主張
[Agent Bの要約: 3-5行]

### 論点の整理

| 観点 | アプローチA | アプローチB | 判定 |
|------|-----------|-----------|------|
| 技術的成熟度 | | | A/B/引分 |
| 実装コスト | | | A/B/引分 |
| 長期保守性 | | | A/B/引分 |
| チーム適合性 | | | A/B/引分 |
| リスク | | | A/B/引分 |

### 統合判断

**推奨**: [A or B or ハイブリッド]

**根拠**: [なぜこの判断に至ったか、両者の議論のどの点が決め手になったか]

**リスク軽減策**: [推奨案のリスクに対する対策]

### Next Steps
1. [具体的なアクション1]
2. [具体的なアクション2]
3. [具体的なアクション3]
```

## 適用例

- 技術選定: 「React vs Vue」「PostgreSQL vs MongoDB」
- アーキテクチャ: 「マイクロサービス vs モノリス」「サーバーレス vs コンテナ」
- 事業判断: 「SES拡大 vs 自社プロダクト注力」「新規採用 vs 外注」
- 投資判断: 「内製 vs SaaS導入」「リプレイス vs 改修」

## Troubleshooting

### 両者の議論が似通ってしまう場合
Agent Bに「Devil's Advocate（悪魔の代弁者）として、意図的に反対の立場を取ってください」と強調する。

### 判断テーマが二択に収まらない場合
3つ以上の選択肢がある場合は、まず2つに絞ってから実行。または、2回実行してトーナメント形式で比較。
