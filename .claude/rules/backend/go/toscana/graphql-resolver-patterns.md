# GraphQL リゾルバー実装パターン

## 0. 大原則（Do / Don't）

- Resolver は入出力の整形に徹する（認証、Node ID 変換、事前ロード、Handler 呼び出し、Presenter 変換、エラー変換）。
- ビジネスロジック、永続化、複雑な条件分岐は Resolver に置かず、Usecase / Domain に寄せる。
- N+1 を避けるため、可能な限り DataLoader を使う。
- エラーは `utils` / `errors` の既存パターンに寄せ、入力 ID の露出有無を意識して使い分ける。
- このリポジトリではログメッセージと（手書きの）コメントは日本語の常体で書く。

## 1. ファイル構造

リゾルバーファイルは gqlgen により自動生成される。手動でのコード変更は生成時に保持される。

```go
package graph

// This file will be automatically regenerated based on the schema, any resolver
// implementations will be copied through when generating and any unknown code will be moved to the end.
// Code generated by github.com/99designs/gqlgen version v0.17.86

import (
    "context"
    // 必要なインポート
)
```

## 2. 認証パターン

### 2.1 ワークスペースユーザー認証（標準パターン）

```go
currentUser, ok := auth.GetWorkspaceUserFromContext(ctx)
if !ok {
    return nil, utils.NewUnauthenticatedError(ctx, msg.WorkspaceUserUnauthenticated)
}
```

### 2.2 ワークスペースユーザーまたはオペレーター認証

```go
currentUser, operator, ok := auth.GetWorkspaceUserOrOperatorFromContext(ctx)
if !ok {
    return nil, utils.NewUnauthenticatedError(ctx, msg.Unauthenticated)
}
```

### 2.3 オプショナル認証（認証なしでも処理継続）

フィールドリゾルバーで認証がオプショナルな場合：

```go
currentUser, ok := auth.GetWorkspaceUserFromContext(ctx)
if !ok {
    return nil, nil  // エラーではなく nil を返す
}
```

注意：このパターンはフィールドが nullable の場合に適する。フィールドが Non-Null（`!`）の場合、`nil` は GraphQL エラーになり得るため、スキーマ設計（nullable 化）や返却値（空配列/空 Connection など）を仕様として決めてから実装する。

## 3. GraphQL Node ID 変換パターン

### 3.1 単一 ID の変換

```go
channelID, err := utils.ParseNodeId(input.ChannelID, enum.NodeChannel)
if err != nil {
    return nil, utils.WrapParseNodeError(ctx, err)
}
```

### 3.2 複数 ID の変換

```go
workspaceUserIDs := make([]string, 0, len(input.WorkspaceUserIds))
for _, id := range input.WorkspaceUserIds {
    workspaceUserID, err := utils.ParseNodeId(id, enum.NodeWorkspaceUser)
    if err != nil {
        return nil, utils.WrapParseNodeError(ctx, err)
    }
    workspaceUserIDs = append(workspaceUserIDs, workspaceUserID)
}
workspaceUserIDs = lo.Uniq(workspaceUserIDs)
```

### 3.3 ページネーション用カーソル変換

```go
if after != nil {
    id, err := utils.ParseNodeId(*after, enum.NodeChannel)
    if err != nil {
        return nil, utils.WrapParseNodeError(ctx, err)
    }
    after = &id
}
if before != nil {
    id, err := utils.ParseNodeId(*before, enum.NodeChannel)
    if err != nil {
        return nil, utils.WrapParseNodeError(ctx, err)
    }
    before = &id
}
```

## 4. DataLoader パターン

### 4.1 単一エンティティのロード

```go
loaders := dataloader.GetLoadersFromContext(ctx)
entity, err := loaders.Channel.Load(ctx, channelID)
if err != nil {
    return nil, err
}
if entity == nil {
    return nil, utils.NewNotFoundError(ctx, input.ChannelID)
}
input.Channel = entity
```

### 4.2 並列ロード（Thunk パターン）

独立した複数のロードを並列実行する：

```go
loaders := dataloader.GetLoadersFromContext(ctx)

{
    t1 := loaders.Channel.LoadThunk(ctx, channelID)
    t2 := loaders.WorkspaceUser.LoadThunk(ctx, workspaceUserID)

    if r, err := t1(); err != nil {
        return nil, err
    } else if r == nil {
        return nil, utils.NewNotFoundError(ctx, input.ChannelID)
    } else {
        input.Channel = r
    }

    if r, err := t2(); err != nil {
        return nil, err
    } else if r == nil {
        return nil, utils.NewNotFoundError(ctx, input.WorkspaceUserID)
    } else {
        input.WorkspaceUser = r
    }
}
```

### 4.3 依存関係のあるロード（2段階バッチ）

第1バッチの結果に依存するロードがある場合：

```go
loaders := dataloader.GetLoadersFromContext(ctx)

{
    // 第1バッチ: eventID ベースの独立したロード
    t1 := loaders.CalendarEvent.LoadThunk(ctx, eventID)
    t2 := loaders.CalendarRecurrence.LoadThunk(ctx, eventID)

    event, err := t1()
    if err != nil {
        return nil, err
    }
    if event == nil {
        return nil, utils.NewNotFoundError(ctx, input.EventID)
    }
    input.Event = event

    recurrence, err := t2()
    if err != nil {
        return nil, err
    }
    input.Recurrence = recurrence

    // 第2バッチ: event.CalendarID に依存するロード
    t3 := loaders.Calendar.LoadThunk(ctx, event.CalendarID)
    calendarShareKey := event.CalendarID + ":" + currentUser.ID
    t4 := loaders.CalendarShare.LoadThunk(ctx, calendarShareKey)

    calendar, err := t3()
    if err != nil {
        return nil, err
    }
    if calendar == nil {
        return nil, utils.NewNotFoundErrorWithMessage(ctx)
    }
    input.Calendar = calendar

    // CalendarShare は存在しない場合もあるため、エラーは無視
    if calendarShare, err := t4(); err == nil {
        input.CalendarShare = calendarShare
    }
}
```

### 4.4 複数エンティティのロード

```go
loaders := dataloader.GetLoadersFromContext(ctx)
entities, err := loaders.WorkspaceUser.LoadAll(ctx, workspaceUserIDs)
if err != nil {
    return nil, err
}
for i, u := range entities {
    if u == nil {
        return nil, utils.NewNotFoundError(ctx, workspaceUserIDs[i])
    }
}
input.WorkspaceUsers = entities
```

## 5. Mutation パターン

### 5.0 事前ロード結果を input に埋め込む

このプロジェクトでは、Go 側の `graph/model` の input 構造体に「内部用フィールド」（例：`*model.Channel` など）を持たせる場合がある。

- Resolver で DataLoader によりエンティティをロードして input に詰める
- Handler / Usecase は「ロード済み」を前提に処理し、未ロード（`nil`）は NotFound / Forbidden 等の既存エラーに落とす

この運用により、Usecase 側で再取得を繰り返さずに済み、責務（取得は Resolver、業務処理は Usecase）も保ちやすくなる。

### 5.1 標準的な Mutation 実装

```go
func (r *mutationResolver) UpdateChannel(ctx context.Context, input model.UpdateChannelInput) (*model.UpdateChannelPayload, error) {
    // 1. 認証
    currentUser, ok := auth.GetWorkspaceUserFromContext(ctx)
    if !ok {
        return nil, utils.NewUnauthenticatedError(ctx, msg.WorkspaceUserUnauthenticated)
    }

    // 2. Node ID 変換
    channelID, err := utils.ParseNodeId(input.ChannelID, enum.NodeChannel)
    if err != nil {
        return nil, utils.WrapParseNodeError(ctx, err)
    }

    // 3. DataLoader で事前ロード
    loaders := dataloader.GetLoadersFromContext(ctx)
    if r, err := loaders.Channel.Load(ctx, channelID); err != nil {
        return nil, err
    } else if r == nil {
        return nil, utils.NewNotFoundError(ctx, input.ChannelID)
    } else {
        input.Channel = r
    }

    // 4. Handler 実行
    channel, err := r.UpdateChannelHandler.Handle(ctx, currentUser, &input)
    if err != nil {
        return nil, err
    }

    // 5. Presenter で変換して返却
    return &model.UpdateChannelPayload{
        ClientMutationID: input.ClientMutationID,
        Channel:          r.ChannelPresenter.Present(channel),
    }, nil
}
```

## 6. Field Resolver パターン

### 6.1 親エンティティの解決（共通ヘルパー使用）

`resolver_helpers.go` に定義された共通ヘルパーを使用する：

- `resolveWorkspaceByID` - ワークスペース解決
- `resolveCalendarByID` - カレンダー解決
- `resolveChannelByID` - チャンネル解決
- `resolveWorkspaceUserByID` - ワークスペースユーザー解決
- `resolveChannelMessageByID` - メッセージ解決

これらは DataLoader + Presenter パターンを内包している。

```go
func (r *channelResolver) Workspace(ctx context.Context, obj *model.Channel) (*model.Workspace, error) {
    return resolveWorkspaceByID(ctx, r.Resolver, obj.WorkspaceID)
}
```

### 6.2 DataLoader を使用したフィールド解決

```go
func (r *channelResolver) TotalMembers(ctx context.Context, obj *model.Channel) (int, error) {
    loaders := dataloader.GetLoadersFromContext(ctx)
    n, err := loaders.ChannelUserCount.Load(ctx, obj.RawID)
    if err != nil {
        return 0, err
    }
    return n, nil
}
```

### 6.3 現在のユーザーに依存するフィールド

```go
func (r *channelResolver) IsUnread(ctx context.Context, obj *model.Channel) (bool, error) {
    currentUser, ok := auth.GetWorkspaceUserFromContext(ctx)
    if !ok {
        return false, nil  // 認証なしは false を返す
    }
    loaders := dataloader.GetLoadersFromContext(ctx)
    isUnread, err := loaders.ChannelIsUnread.Load(ctx, currentUser.ID+":"+obj.RawID)
    if err != nil {
        return false, err
    }
    return isUnread, nil
}
```

## 7. Connection (Pagination) パターン

### 7.0 引数バリデーション指針

- `first` と `last` の同時指定は仕様として禁止する（どちらか片方のみを許可する）。
- `first` / `last` には上限を設ける（過大な要求による負荷増大を防ぐ）。
- 0 や負数など不正値は早期にエラーにする（もしくはデフォルト値に丸める）。
- `after` / `before` は対応する方向（forward/backward）と整合するように扱う。

```go
func (r *workspaceResolver) Channels(ctx context.Context, obj *model.Workspace, filtered *bool, first *int, after *string, last *int, before *string) (*model.ChannelConnection, error) {
    // デフォルト値の設定
    if filtered == nil {
        filtered = new(bool)
        *filtered = true
    }

    // カーソルの変換
    if after != nil {
        id, err := utils.ParseNodeId(*after, enum.NodeChannel)
        if err != nil {
            return nil, utils.WrapParseNodeError(ctx, err)
        }
        after = &id
    }
    if before != nil {
        id, err := utils.ParseNodeId(*before, enum.NodeChannel)
        if err != nil {
            return nil, utils.WrapParseNodeError(ctx, err)
        }
        before = &id
    }

    // Handler 実行
    output, err := r.FetchChannelsHandler.Handle(ctx, obj.RawID, *filtered, first, after, last, before)
    if err != nil {
        return nil, err
    }

    // Presenter で Connection を構築
    return r.ChannelPresenter.PresentConnection(output.Nodes, output.HasNext, output.HasPrev, output.Total), nil
}
```

## 8. エラーハンドリング

### 8.1 エラータイプと使用場面

| 関数                                | 用途                                                    |
| ----------------------------------- | ------------------------------------------------------- |
| `utils.NewUnauthenticatedError`     | 認証が必要だが認証されていない                          |
| `utils.NewForbiddenError`           | 認証済みだが権限がない                                  |
| `utils.NewNotFoundError`            | ユーザー入力の ID でエンティティが見つからない          |
| `utils.NewNotFoundErrorWithMessage` | 内部 ID でエンティティが見つからない（ID を露出しない） |
| `utils.WrapParseNodeError`          | Node ID のパースエラー                                  |

### 8.2 エラー使い分けの原則

```go
// ユーザー入力の ID → NewNotFoundError（ID を含むメッセージ）
if r == nil {
    return nil, utils.NewNotFoundError(ctx, input.ChannelID)
}

// 内部 ID（ロード済みオブジェクトから導出）→ NewNotFoundErrorWithMessage
if calendar == nil {
    return nil, utils.NewNotFoundErrorWithMessage(ctx)
}
```

## 9. 実装チェックリスト / アンチパターン

### 9.1 実装チェックリスト

- [ ] 認証（必須/オプショナル）を仕様どおりに扱っている
- [ ] Node ID は必ず `utils.ParseNodeId` で変換し、パースエラーは `utils.WrapParseNodeError` に寄せている
- [ ] 取得は DataLoader を優先し、N+1 を作っていない
- [ ] 事前ロードが必要なら input の内部用フィールドへ詰めて Handler / Usecase に渡している
- [ ] Presenter で `model`（GraphQL）へ変換して返している
- [ ] NotFound のメッセージ露出（入力 ID か/内部 ID か）を `NewNotFoundError` と `NewNotFoundErrorWithMessage` で使い分けている

### 9.2 よくあるアンチパターン

- Resolver でリポジトリを直接叩き、N+1 を作る
- Node ID の変換を省略して DB の生 ID と混在させる
- `NewNotFoundErrorWithMessage` を使うべき箇所で入力 ID を露出する
- Resolver でトランザクションを開始し、業務処理まで抱え込む（Tx は Usecase へ）
- 仕様が未確定のまま `nil, nil` を返してしまい、Non-Null フィールドで GraphQL エラー化する
