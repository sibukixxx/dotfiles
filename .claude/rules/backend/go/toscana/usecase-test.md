# ユースケース層のテスト (`internal/usecase/*_test.go`)

このドキュメントは、`internal/usecase` ディレクトリ配下にあるユースケース層のテストコードの書き方を説明する。

テストコードの基本的な規約（命名規則、アサーション、モックなど）は、`docs/testing_rules.md` を参照する。

## テストの目的

ユースケース層のテストは、以下の目的で行う：

- **ビジネスロジックの検証**: ユースケース固有の処理フローや条件分岐が正しく実装されていることを確認する
- **依存コンポーネントとの連携確認**: リポジトリ層、サービス層など、依存するコンポーネントとのインタラクションが期待通りに行われることを確認する
- **権限チェックの検証**: 操作を実行するユーザーが必要な権限を持っているかどうかのチェックが正しく機能することを確認する
- **入力バリデーションの検証**: 不正な入力値に対するバリデーションロジックとエラーハンドリングが適切であることを確認する
- **監査ログの記録確認**: 操作が実行された際に、期待される監査ログが記録されることを確認する

## テストの構造

ユースケース層のテストでは、多様な入力と条件分岐を効率的に検証するため、**テーブル駆動テスト**を主体として記述する。詳細な実装方法は `docs/testing_rules.md` の「テスト構造」セクションを参照する。

```go
func TestInviteChannelUserUsecase_Execute(t *testing.T) {
    t.Parallel()

    tests := []struct {
        name    string
        // ...
        setup   func(m *mockContainer) // モックの設定
        // ...
    }{
        {
            name: "正常系: ユーザーをチャンネルに招待できる",
            setup: func(m *mockContainer) {
                // モックの期待値設定
            },
            // ...
        },
        // ...
    }

    for _, tt := range tests {
        // ...
        t.Run(tt.name, func(t *testing.T) {
            // ...
        })
    }
}
```

### **1. モックの利用**

ユースケースが依存するインターフェースは、`go.uber.org/mock/gomock` を用いてモック化する。各テストケースの `setup` 関数内で、モックの期待値を設定する。

- **テストごとのモックコントローラー**: 各サブテスト (`t.Run`) の中で `gomock.NewController(t)` を生成し、テストケース間のモックの状態が独立するようにする
- **トランザクション管理**: `repository.TxManager` は、`mock_repository.NewMockTxManagerWithImpl(ctrl)` のように、渡された関数を実際に実行する実装を持つモックを使用し、トランザクション内の処理をシミュレートする

### **2. 権限チェックの検証**

`operator_authz` や `workspace_authz` パッケージで定義された権限チェックロジックが、ユースケース内で適切に呼び出されることを検証する。権限がある場合とない場合の両方のシナリオをテストケースに含める。

### **3. バリデーションの検証**

`service.ValidatorService` を使用した入力値の検証ロジックをテストする。有効な入力と無効な入力に対するユースケースの挙動を確認し、期待されるエラーが返ることを検証する。

### **4. 監査ログの検証**

操作が実行された際に期待される監査ログが記録されるかを検証する。`audit.Logger` のモックが、期待される `audit.AuditEvent` を引数として呼び出されることを確認する。

- **状態遷移の検証**: 変更があるフィールドだけでなく、変更がない（維持されるべき）フィールドも監査ログの期待値に含めて検証し、意図しない変更を防ぐ
- **順序非依存の比較**: `cmpopts.SortSlices` を利用して、監査ログの `Changes` スライスの順序に依存しない比較を行う

### **5. 結果の検証**

ユースケースメソッドの実行結果と期待値を比較する。詳細なアサーション方法は `./testing_rules.md` の「アサーションと比較」セクションを参照する。
