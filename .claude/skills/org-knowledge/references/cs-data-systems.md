# データシステム・分散システム・トランザクション

## データ指向アプリケーション設計（Kleppmann『DDIA』ベース）

### データモデルとクエリ言語

| モデル | 強み | 弱み | 適用 |
|--------|------|------|------|
| リレーショナル | 結合、スキーマ強制、ACID | 柔軟性、水平スケール | 業務系、会計、在庫 |
| ドキュメント | スキーマ柔軟、局所性 | 結合が弱い、多対多 | コンテンツ管理、イベント |
| グラフ | 多対多の関係探索 | 集約が弱い | ソーシャル、知識グラフ |
| 列指向 | 分析クエリ、圧縮効率 | 点クエリが遅い | DWH、OLAP |

### ストレージエンジンの内部構造（Petrov『Database Internals』補強）

#### B-tree vs LSM-tree

```
【B-tree】
├ 構造: バランス木、ページ単位の読み書き
├ 読み取り: O(log n) — ページをルートからリーフへ辿る
├ 書き込み: ページ内更新 + WAL（先行書込みログ）
├ 長所: 読み取り性能が安定、範囲クエリが得意
├ 短所: 書込み増幅（write amplification）
└ 採用例: PostgreSQL, MySQL/InnoDB, SQLite

【LSM-tree（Log-Structured Merge-tree）】
├ 構造: メモリ上のmemtable → ディスク上のSSTable（ソート済み）
├ 書き込み: memtableに追記 → フラッシュ → コンパクション
├ 読み取り: memtable → 各レベルのSSTableを確認（Bloom filterで高速化）
├ 長所: 書込みスループットが高い、シーケンシャルI/O
├ 短所: 読み取り増幅、コンパクションのI/O負荷
└ 採用例: RocksDB, LevelDB, Cassandra, ScyllaDB
```

#### WAL（Write-Ahead Log）

```
全ての変更をまずWALに書き込む → クラッシュ後にWALから復旧

WALの保証:
1. 順序性: 操作の順序を保存
2. 耐久性: fsync でディスクに永続化
3. 原子性: WALのレコード単位で適用/ロールバック

用途: DB復旧、レプリケーション、CDC（Change Data Capture）
```

### レプリケーション

| 方式 | 構成 | 整合性 | レイテンシ |
|------|------|--------|-----------|
| 単一リーダー | 1 writer + N readers | 強い（リーダー経由） | 書込み: リーダーへ |
| マルチリーダー | N writers + N readers | 結果整合（コンフリクト解決必要） | 各リージョンでローカル |
| リーダーレス | 全ノードが読み書き | 定足数（quorum）次第 | 最寄りノード |

#### レプリケーションの遅延問題

```
Read-after-write 整合性: 自分の書込みを必ず読める
  → リーダーから読む / バージョン番号で確認

Monotonic reads: 過去に見えた値より古い値を見ない
  → 同一レプリカから読む

Consistent prefix reads: 因果順序を保つ
  → パーティション内の順序を維持
```

### パーティショニング（シャーディング）

| 方式 | 分割基準 | 長所 | 短所 |
|------|---------|------|------|
| キーレンジ | キーの範囲 | 範囲クエリ効率的 | ホットスポット |
| ハッシュ | キーのハッシュ | 均等分散 | 範囲クエリ不可 |
| 複合 | ハッシュ + レンジ | バランス | 複雑性 |

リバランシング: 固定パーティション数 / 動的分割 / ノード比例

### トランザクション（Gray & Reuter『Transaction Processing』補強）

#### ACID

```
Atomicity: 全部成功 or 全部なかったことに（abort → undo）
Consistency: 不変条件を維持（アプリケーション側の責任）
Isolation: 並行トランザクションが互いに干渉しない
Durability: コミット済みデータは永続化される（WAL + fsync）
```

#### 分離レベル

| レベル | ダーティリード | 非反復読取り | ファントム | 実装例 |
|--------|-------------|------------|----------|--------|
| Read Uncommitted | ○ | ○ | ○ | ほぼ使わない |
| Read Committed | × | ○ | ○ | PostgreSQL default |
| Repeatable Read / Snapshot Isolation | × | × | △ | MySQL InnoDB default |
| Serializable | × | × | × | SSI, 2PL |

#### 並行性制御

```
【2PL（Two-Phase Locking）】
Growing phase: ロックを取得のみ
Shrinking phase: ロックを解放のみ
→ Serializable だがデッドロックの可能性

【MVCC（Multi-Version Concurrency Control）】
各行に複数バージョンを保持
読み取りはロック不要（スナップショットを参照）
→ 読み取りと書込みがブロックし合わない

【SSI（Serializable Snapshot Isolation）】
楽観的: まずスナップショットで実行
コミット時に競合を検出 → 競合があればabort
→ PostgreSQL の serializable はこれ
```

#### 分散トランザクション

```
【2PC（Two-Phase Commit）】
Phase 1 (Prepare): コーディネータが全参加者にprepare要求
Phase 2 (Commit/Abort): 全員OK → commit / 1人でもNG → abort

問題:
- コーディネータ障害 → 参加者がin-doubt状態でブロック
- レイテンシ（2ラウンドトリップ）

【Saga パターン】
長時間トランザクションを補償トランザクションの連鎖で実現
T1 → T2 → T3（失敗）→ C2 → C1（補償）
→ 結果整合性で十分な場合に使う
```

### 一貫性と合意

#### CAPの定理（正確な理解）

```
Consistency（線形化可能性）、Availability、Partition tolerance の3つのうち
ネットワーク分断時に2つしか同時に満たせない

実際の選択: ネットワーク分断は避けられないので CP or AP
  CP: 分断時に一部のリクエストを拒否（整合性優先）
  AP: 分断時も応答するが古いデータの可能性（可用性優先）
```

#### 合意アルゴリズム

| アルゴリズム | 特徴 |
|-------------|------|
| Paxos | 理論的基盤。理解が難しい |
| Raft | Paxosの理解しやすい変種。リーダー選出 + ログ複製 |
| ZAB | ZooKeeperで使用 |
| PBFT | ビザンチン障害耐性（悪意あるノード対応） |

### ストリーム処理

```
【イベントソーシング】
状態を直接更新せず、イベント（事実の記録）を追記
現在の状態はイベントを再生して導出

利点: 監査証跡、時間旅行、イベント駆動アーキテクチャ
課題: スナップショット管理、イベントスキーマ進化

【CDC（Change Data Capture）】
DBの変更をイベントストリームとして配信
WAL → Kafka → ダウンストリーム（検索インデックス、キャッシュ、DWH）

【ストリーム処理のセマンティクス】
at-most-once: 最大1回処理（失われる可能性）
at-least-once: 少なくとも1回（重複の可能性）→ 冪等性で対処
exactly-once: 正確に1回（分散では困難 → 冪等性 + トランザクションで近似）
```

### バッチ処理 vs ストリーム処理

| 特性 | バッチ | ストリーム |
|------|--------|-----------|
| 入力 | 有限（ファイル） | 無限（イベント） |
| レイテンシ | 分〜時間 | ミリ秒〜秒 |
| 出力 | 新しいデータセット | イベント or 状態更新 |
| 障害対応 | 再実行 | チェックポイント + リプレイ |
| 例 | MapReduce, Spark | Kafka Streams, Flink |

## エージェント基盤への応用

| CS概念 | エージェント開発での活用 |
|--------|----------------------|
| WAL | エージェントの状態変更ログ、クラッシュリカバリ |
| イベントソーシング | エージェントの行動履歴、監査証跡 |
| 冪等性 | ツール呼び出しの再実行安全性 |

| パーティショニング | 大規模ベクトルDB、ジョブの分散処理 |
| 合意 | リーダー選出（ワーカー調整）、設定の一貫性 |
| ストリーム処理 | リアルタイムイベント駆動ワークフロー |
