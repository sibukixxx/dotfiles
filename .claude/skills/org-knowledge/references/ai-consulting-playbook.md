# AI導入プレイブック

## 1. AI案件の進め方

1. 業務課題を分解する
2. AIが効く作業を特定する
3. KPIとベースラインを置く
4. PoCの成功条件を定義する
5. 本番運用の制約を確認する

## 2. ユースケース選定の観点

| 観点 | みるポイント |
|------|-------------|
| 価値 | 工数削減、品質向上、売上寄与、リスク低減 |
| 実現性 | データ取得可否、精度要件、業務への組み込みやすさ |
| 安全性 | 機密性、誤回答許容度、監査要求 |
| 運用性 | 更新頻度、担当部門、継続改善体制 |

## 3. RAGの設計観点

- 検索単位は「回答に必要な意味のまとまり」で切る
- チャンクサイズは精度と回収率の両方で評価する
- メタデータで絞り込み可能にする
- 再ランキングの有無を比較する
- 回答不能時の挙動を設計する

## 4. エージェント設計観点

- 目的、権限、停止条件を分ける
- 外部ツール呼び出しは監査可能にする
- 冪等性と再実行安全性を考える
- 人手承認が必要な境界を定義する

## 5. 評価

### 最低限の評価軸

- 正答率 / タスク成功率
- 根拠一致率
- 幻覚率
- レイテンシ
- コスト

### 評価データセット

- 代表ケース
- 難ケース
- 失敗が許されないケース
- 禁止すべきケース

## 6. モデル選定

- 推論品質
- 構造化出力の安定性
- コンテキスト長
- 応答速度
- 価格
- データ保持ポリシー
- リージョン / コンプライアンス要件

## 7. 安全性とガバナンス

- PII・機密情報の送信制御
- プロンプトインジェクション対策
- 出力フィルタ
- 操作ログ
- フィードバックループ

## 8. PoCの失敗例

- 精度目標が曖昧
- 現場運用への組み込みがない
- ナレッジの整備不足をモデルで隠そうとする
- 評価用データを作らず「なんとなく良い」で判断する
