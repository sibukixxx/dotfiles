# 元記事の批判的レビュー

参照元エッセイ「You are doing everything... AI workflows」に対する技術的検証。

**フレームワーク部分は妥当**。問題は **抜け** と **順番**。

| 観点 | 評価 |
|------|------|
| ワークフロー起点の発想 | ✅ 正しい |
| 5コンポーネント構造 | ✅ 妥当 |

## 具体的な誤り・誇張

### 1. 「AI does not sleep, does not forget」

- ❌ Sleep しない → API は rate limit や outage で実質止まる
- ❌ Forget しない → コンテキスト上限で忘れる、モデル更新で挙動変わる
- ❌ Get sick しない → ベンダーの障害は普通にある（Claude / OpenAI 共に）

正確な表現: 「AI は決まった処理を反復的にこなすのが得意」。それ以上は誇張。

### 2. 「60-70% of workflow steps are automatable」

- 業種・業務によって 20% の業界もあれば 80% の業界もある
- 何の根拠も提示されていない
- 「自分の業務を棚卸しすれば分かる」が誠実な答え

### 3. 「Hiring is expensive. Training takes months」と AI を対比

- AI 構築も訓練・運用コストがかかる
- プロンプトエンジニアリング・モニタリング・改善で **継続的な工数** が発生する
- 「人件費 vs AI コスト」の比較を冷静にする必要

### 4. Email Operations Center を Workflow 1 として推す

- 顧客対面で誤動作の影響が大きい
- 分類タスクは曖昧（「sales inquiry」と「support」の境界が不明確なメールが多い）
- 最初に作ると **誤分類事故で AI 自動化への信頼を失う**

→ 本スキルでは **Report Factory を最初** に推奨。

### 5. 「Quality check: 10% sample weekly」

- 弱すぎる。最初は 100% レビュー、誤動作率が KPI を下回ってから段階的に下げる
- 顧客対面ワークフローで 10% サンプリングは事故を見逃す

### 6. 「Workflow 1-2-3 を Week 3-6 で構築」

- 1 ワークフロー = 2-3 週間が現実（テスト・検証・shadow mode を含めて）
- 3 ワークフローで 4 週間は専任エンジニアでもキツい
- 副業 / 兼任なら 3-6 ヶ月

### 7. 「Use n8n, Zapier, or Make」（選定基準なし）

- これらは **目的が異なる**：
  - **Zapier**: 早く作れるが、ロジックが重くなると限界
  - **Make**: ビジュアルでロジック組める、Zapier より柔軟
  - **n8n**: OSS / セルフホスト可、最も柔軟だが運用コスト
  - **自前コード**: 最大の柔軟性、開発・運用コスト最大
- 選定基準を提示すべき（→ orchestrator-selection.md で補完）

### 8. 「Connect workflows」の説明が曖昧

- 「event system」とだけ言って具体性なし
- イベント駆動の設計は重要なテーマだが、broker（Redis / SQS / EventBridge）選定や schema 設計が欠落
- 結局のところ Zapier / Make の中で connection を作る話か、独立した event bus を立てる話かが不明

### 9. 「The businesses that build this system now will operate at a fundamentally different level」

- 煽り文。実際には：
  - 構築だけで終わって運用できないチームが多数
  - 本質は **継続改善できるチーム** が差をつける、ツールではない
  - 自動化された粗悪な業務は、手作業の高品質な業務に勝てない場合もある

### 10. ROI / 失敗リスクへの言及がない

- 構築失敗・運用失敗で投資回収できないケースが多い
- 「自動化したものの誰も使わない」「壊れて人手戻し」の事例が一切議論されていない

## 推奨される読み方

このエッセイは **マーケティング素材** として読み、技術的な実装ガイドとしては不十分。フレームワーク（5 phases / 5 components）は使えるが、以下を必ず追加：

1. **観測性** を最初の workflow から組み込む（→ missing-pieces.md）
2. **テスト** をプロンプトと workflow 両方で書く（→ missing-pieces.md）
3. **セキュリティ** の最低ラインを満たす（→ missing-pieces.md）
4. **コスト** に上限と監視を設定する（→ missing-pieces.md）
5. **Report Factory から始める**（→ workflow-templates.md）
6. **段階的解禁** で顧客対面ワークフローに進む（→ workflow-templates.md）
7. **タイムラインを倍に取る**（楽観バイアス補正）

## 一行サマリ

> 「ワークフロー起点で考える」のは正しい。だが「自動化すれば何とかなる」は誤り。**観測・テスト・セキュリティ・コスト管理** を組み込まない自動化は、人間の手作業より早く事故を起こす。
