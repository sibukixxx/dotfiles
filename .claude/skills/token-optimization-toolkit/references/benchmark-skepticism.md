# 削減率の主張をどう読むか（懐疑ガイド）

## なぜ「97%削減」「98%削減」が並ぶのか

これらのツールが提示する数字は、ほぼ全て **特定タスク・特定条件下のチャンピオンデータ** である。一般化はできない。

### 数字の典型的トリック

1. **削減対象を限定**
   - 「ツール出力経由のトークンのみ」（Context Mode の 98%）
   - 「コードナビ経由のトークンのみ」（Token Savior の 97%）
   - 「出力（assistant message）トークンのみ」（Caveman の 75%、Claude Token Efficient の 63%）

   コーディングエージェントの実コストはおおむね **入力 + キャッシュ読み込み** が支配的で、これら部分削減の寄与は限定的。

2. **キャッシュヒット前提**
   - ooples Token Optimizer MCP の「95%+」は **キャッシュヒット時** の数字。初回・コールドキャッシュでは効果ゼロ。

3. **チェリーピックされたタスク**
   - Code Review Graph の「49x」は **特定の変更影響解析タスク**。普通の Q&A や調査では出ない。
   - nadimtuhin の「11K → 1.3K」は **「初期コンテキストとして読まれるドキュメントトークン」のみ** を比較。実セッションのトークン消費全体ではない。

4. **比較対象が無設定状態**
   - 「Vanilla の Claude Code 設定」と比較した削減率は、既にチューニング済みの環境への追加効果を表さない。
   - 既存 `.claude/CLAUDE.md` でチューニング済みなら、追加効果は半減することが多い。

5. **トークン数 ≠ コスト ≠ レイテンシ**
   - 入力トークンは prompt cache でほぼ無料。output トークンの方が単価が高い。
   - 大量の small chunks 読み込みより、1 つの大きな file 読み込みの方が cache に乗りやすい場合がある。
   - 「トークン削減 = コスト削減」と直結しない。

## 自分のプロジェクトで検証する手順

### ステップ 1: ベースライン測定

```bash
# 何も入れない状態で代表タスク 5 個を実行
# 1 タスクあたり以下を記録
- /cost の output（input/output/cache_read/cache_write トークン）
- 経過時間
- ユーザー満足度（タスク完了したか、品質）
```

代表タスクの例:
- 既存ファイルへの機能追加
- バグ修正（ファイル特定 + 修正）
- 大規模リファクタリング
- コードレビュー
- ドキュメント生成

### ステップ 2: ツール導入

レシピ通り 1 つだけ導入する。複数同時はやらない（効果切り分け不能）。

### ステップ 3: 同じタスクを再実行

- なるべく同じ条件（同じファイル、同じプロンプト）
- セッションを毎回新規にする（cache の影響を排除）
- 5 タスク × 各 3 回 = 15 サンプルあれば最低限の傾向は見える

### ステップ 4: 比較

| 指標 | ベースライン | ツール導入後 | 削減率 |
|------|------------|------------|-------|
| input tokens | – | – | – |
| output tokens | – | – | – |
| cache_read tokens | – | – | – |
| 合計コスト USD | – | – | – |
| 経過時間 | – | – | – |
| タスク成功率 | – | – | – |

**重要**: トークン削減で **タスク成功率が下がっている** なら、その削減は意味がない。

### ステップ 5: 判断基準

- **コスト削減 < 20%** で **品質低下なし**: 維持してもよいが優先度低
- **コスト削減 ≥ 30%** で **品質低下なし**: 採用
- **コスト削減 ≥ 30%** だが **タスク失敗増**: 採用しない、もしくは限定的な使い方に留める
- **コスト削減 < 10%**: ノイズ。撤去

## 注意すべきアンチパターン

### 「とりあえず全部入れる」

- ツール同士の競合で予期しない挙動
- 設定ファイルが肥大化、本末転倒
- どれが効いているか測定不能

### 「ベンダーの数字をそのまま顧客／チームに伝える」

- 期待値が過度に高くなる
- 実数値で下回ったときに信用を失う

### 「測定なしで導入を続ける」

- 効果を確認せずに新ツールを次々追加 → 元より遅い・高い状況になる
- 撤去基準も持てない

### 「短期セッションで Caveman 系を評価」

- 出力スタイル系は **長時間・大量出力** が前提
- 短いタスクでは benefit がほぼ出ず、見栄えだけ悪くなる

## 健全な期待値

実プロジェクトで本気で測ると、**個別ツールの寄与は 10–30% 程度** が現実的なレンジ。

- 50%超の削減は、よほど無駄が多い初期状態か、特定タスクに限定された場合のみ
- 70%超の主張は、ほぼ全てが「対象を絞った数字」と思って良い

ただし、複数の **直交する** 改善（CLI 出力圧縮 + コードベース検索 + CLAUDE.md 整備）を組み合わせれば、合計で 40–60% 削減は射程に入る。
