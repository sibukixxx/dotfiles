# コンパウンドナレッジの原理

## ナレッジが最も濃いリポジトリで作業する

これが最も重要な原則。

### なぜか

Claude Code のコンテキストは、セッション開始時に以下を自動で読み込む:

1. `.claude/CLAUDE.md` - プロジェクトルール
2. `.claude/rules/` - ドメインルール
3. `.claude/skills/` - スキル（トリガー時）
4. `.claude/commands/` - カスタムコマンド

**ナレッジが濃いリポジトリ = これらのファイルが充実しているリポジトリ**

このリポジトリで作業を開始すれば、すべてのナレッジが自動的にコンテキストに入る。
そこから `../other-repo` を参照・編集すれば、ナレッジを活用しながら別リポジトリで作業できる。

### コンパウンド効果

```
セッション1: スキルA を作成 → 次回から自動利用
セッション2: ルールB を追加 → スキルA + ルールB が効く
セッション3: リファレンスC を蓄積 → A + B + C が効く
...
セッションN: すべてが複合的に効いて、作業速度が指数関数的に向上
```

### モノレポとの違い

| 観点 | モノレポ | Cross-Repo Knowledge |
|------|---------|---------------------|
| リポジトリ構成 | 単一リポジトリ | 独立した複数リポジトリ |
| ナレッジの所在 | リポジトリ内 | ナレッジハブ（このリポジトリ） |
| 適用範囲 | リポジトリ内のみ | 任意の兄弟リポジトリ |
| OSS対応 | 内部情報も公開 | 内部情報は分離可能 |
| 他者リポジトリ | 不可能 | 参照から読み書きまで対応 |
| 機密情報 | 分離困難 | z-ai/ で完全分離 |

### アンチパターン

- ターゲットリポジトリで直接 `claude` を起動する → ナレッジが効かない
- ナレッジをターゲットリポジトリにコピーする → 二重管理になる
- モノレポに統合する → 権限・公開範囲の問題が発生する

### ベストプラクティス

1. **ナレッジはここに集約**: rules, skills, references すべてこのリポジトリに
2. **作業メモは z-ai/ に**: gitignore済みなので安全
3. **ターゲットには成果物だけ**: コード・ドキュメントなど最終成果物のみ
4. **git -C で操作**: ターゲットリポジトリのgit操作はカレントディレクトリを変えずに
