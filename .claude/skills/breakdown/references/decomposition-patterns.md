# 分解パターン集

タスクの種類別に、25分以内のサブタスクへの分解パターンを示す。

## 1. 開発タスク

### 新機能実装

```
元: 「ユーザープロフィール機能を作る」

分解:
□ データモデル設計（スキーマを書く） - 15min
□ マイグレーションファイル作成 - 10min
□ APIエンドポイント: GET /profile - 20min
□ APIエンドポイント: PUT /profile - 20min
□ フロント: プロフィール表示コンポーネント - 25min
□ フロント: 編集フォームコンポーネント - 25min
□ API連携: 表示機能 - 15min
□ API連携: 更新機能 - 15min
□ エラーハンドリング追加 - 20min
□ 手動テスト - 15min
```

### リファクタリング

```
元: 「Userクラスをリファクタリングする」

分解:
□ 現状のメソッド一覧を作成 - 10min
□ 責務を整理（図に書く） - 15min
□ 分割後のクラス構成を決める - 15min
□ テストを書く（既存の振る舞いを保証） - 25min
□ クラスAを抽出する - 20min
□ クラスBを抽出する - 20min
□ 元クラスから委譲する形に変更 - 15min
□ テストが通ることを確認 - 10min
```

### バグ修正

```
元: 「購入処理でエラーが発生する」

分解:
□ バグ再現手順を文書化 - 10min
□ ログを確認して原因を絞る - 20min
□ デバッガで該当箇所を特定 - 25min
□ 修正方法を2案考える - 15min
□ 修正を実装 - 20min
□ 再現手順で確認 - 10min
□ エッジケースをテスト - 15min
□ PRを作成 - 10min
```

## 2. 調査・分析タスク

### 技術調査

```
元: 「React状態管理ライブラリを比較する」

分解:
□ 現在の課題を3つ書き出す - 10min
□ 候補ライブラリを5つリストアップ - 15min
□ 比較軸を決める（学習コスト、パフォーマンス等） - 10min
□ 各ライブラリのREADMEを読む - 20min
□ 1つ目をサンプル実装 - 25min
□ 2つ目をサンプル実装 - 25min
□ 比較表を作成 - 15min
□ 推奨を決めて理由を書く - 15min
```

### 障害調査

```
元: 「本番環境でレスポンスが遅い」

分解:
□ 発生時刻と影響範囲を確認 - 10min
□ メトリクス（CPU、メモリ、DB）を確認 - 15min
□ 該当時間のログを確認 - 20min
□ 仮説を3つ立てる - 10min
□ 仮説1を検証 - 25min
□ 仮説2を検証 - 25min
□ 原因と対策を文書化 - 20min
```

## 3. ドキュメントタスク

### 設計ドキュメント

```
元: 「システム設計書を書く」

分解:
□ 目次（アウトライン）を書く - 10min
□ 背景・目的セクション - 20min
□ スコープ（やること・やらないこと） - 15min
□ 全体アーキテクチャ図を描く - 25min
□ コンポーネント説明を書く - 25min
□ データフロー図を描く - 20min
□ 非機能要件を書く - 15min
□ セルフレビュー - 15min
```

### 手順書

```
元: 「デプロイ手順書を作る」

分解:
□ 現在の手順をメモ書きで列挙 - 15min
□ 前提条件セクションを書く - 10min
□ 手順を整形（番号付きリスト） - 20min
□ 各手順にコマンド例を追加 - 20min
□ トラブルシューティングセクション - 20min
□ 実際に手順通りやってみて確認 - 25min
□ 不足を追記 - 15min
```

元: 「新機能の企画書を書く」

□ 課題を1文で書く - 5min

□ 実現イメージ（ワイヤーフレーム） - 25min
□ 工数見積もり - 15min

### レビュー依頼

```
元: 「PRレビューを依頼する」

分解:
□ 変更内容を3行でまとめる - 10min
□ テスト手順を書く - 15min
□ 注目してほしいポイントを書く - 10min
□ 関連チケット/ドキュメントをリンク - 5min
□ レビュワーにメンション - 5min
```

## 5. 学習タスク

## 分解のコツ

### 動詞を具体的にする

| 曖昧 | 具体的 |
|------|--------|
| 検討する | 選択肢を3つ書き出す |
| 調べる | READMEを読んでメモを取る |
| 実装する | 関数Xを書く |
| 確認する | テストを実行して結果を記録 |
| 相談する | Slackで質問を投稿する |

### 完了条件を明確にする

```
✗ 「設計を考える」
✓ 「設計案をNotionに1ページ書く」

✗ 「ライブラリを調べる」
✓ 「3つのライブラリの比較表をSpreadsheetに作る」

✗ 「コードを書く」
✓ 「UserServiceのcreateメソッドを実装してテストを通す」
```

### 依存関係を意識する

```
並列可能:
□ コンポーネントA作成
□ コンポーネントB作成
  → 両方同じセッションで進められる

直列のみ:
□ APIを作成 → □ フロントでAPIを呼ぶ
  → APIが完成するまでフロントは着手不可
  → 「APIのモック作成」を間に挟む
```
