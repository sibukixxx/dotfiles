# タスク見積もりガイド

25分タスクへの分解を支援するための見積もり指針。

## 見積もりの基本

### 時間の目安

| 時間 | 適したタスク |
|------|-------------|
| **5分** | 単純な確認、コピペ、コマンド1つ |
| **10分** | 小さな修正、短いメモ作成 |
| **15分** | 小規模な実装、セクション1つの執筆 |
| **20分** | 中規模な実装、調査のまとめ |
| **25分** | 1機能の実装、1つの図の作成 |

### 25分を超えたら分解

```
目安: 25分を超えそうなら分解する

30分のタスク → 15min + 15min に分解
45分のタスク → 25min + 20min に分解
60分のタスク → 25min + 20min + 15min に分解
```

## 見積もりの補正

### 慣れ度による補正

| 慣れ度 | 補正 |
|--------|------|
| 初めてやる | x2.0 |
| 数回やった | x1.5 |
| 慣れている | x1.0 |
| 熟練 | x0.8 |

```
例: 「関数1つ書く」= 15min の場合

初めての言語 → 15min x 2.0 = 30min → 分解が必要
慣れた言語   → 15min x 1.0 = 15min → そのまま
```

### 不確実性による補正

| 不確実性 | 補正 | 対応 |
|----------|------|------|
| 明確 | x1.0 | そのまま |
| 少し曖昧 | x1.3 | 確認タスクを追加 |
| 曖昧 | x1.5 | 調査タスクを先に |
| 不明確 | x2.0 | まず範囲を決めるタスク |

```
例: 「APIを実装する」= 20min の場合

仕様明確     → 20min → そのまま
仕様少し曖昧 → 26min → 「仕様確認」10min + 「実装」20min
仕様不明確   → 40min → 「仕様整理」15min + 「実装」25min
```

## よくある見積もりミス

### 過小見積もりパターン

| ミス | 原因 | 対策 |
|------|------|------|
| 「すぐ終わる」 | 楽観バイアス | 過去の実績で補正 |
| 「コピペでOK」 | 調整を忘れる | 調整時間を+10min |
| 「前やったから」 | 環境差を忘れる | 確認時間を+5min |
| 「簡単なはず」 | 未知の罠 | バッファ+20% |

### 過大見積もりパターン

| ミス | 原因 | 対策 |
|------|------|------|
| 「念のため多めに」 | 不安 | 具体的に分解する |
| 「全部やる」 | スコープ過大 | 最小限に絞る |
| 「完璧にしたい」 | 品質過剰 | v0.1を目指す |

## タイムボックス活用

### ポモドーロ・テクニック

```
25分作業 → 5分休憩 → 25分作業 → 5分休憩 → ...
4ポモドーロ後は15-30分休憩
```

### タイムボックスの使い方

**固定時間で区切る:**

```
「調査は25分まで」と決める
  → 25分で終わらなくても一度止める
  → 得られた情報で次のアクションを決める
```

**見積もり不能タスクへの対応:**

```
「この問題を調査する」（時間読めない）

→ タイムボックス化:
  □ 25分で調査する
  □ わかったことをメモする
  □ 次のアクションを決める

→ 結果に応じて追加:
  □ 追加で25分調査する
  or
  □ 詳しい人に聞く（10min）
```

## 見積もりの振り返り

### 記録フォーマット

```markdown
## 見積もり記録

| タスク | 見積 | 実績 | 差分 | 原因 |
|--------|------|------|------|------|
| API実装 | 20min | 35min | +15min | 例外処理忘れ |
| テスト | 15min | 10min | -5min | 慣れた |
```

### 精度向上のコツ

1. **記録する**: 見積もりと実績を残す
2. **振り返る**: 差分の原因を考える
3. **補正する**: 次回に活かす
4. **パターン化**: よくやるタスクは実績ベースで
