# 行政システムモダナイゼーション スキル

## 概要

このスキルは、経済産業省・資源エネルギー庁の「石油情報システム更改」要件定義報告書
（令和4年度、アビームコンサルティング作成）を基に体系化した、**行政統計調査システムの
モダナイゼーション知見集**である。

レガシーな統計調査システム（オンプレミス、FAX/郵送混在、手動運用）を、
クラウドベースのWebアプリケーションへ刷新する際のベストプラクティスを提供する。

## 適用フェーズ

```
現状分析(As-Is) → 課題抽出 → 施策立案 → 将来像設計(To-Be) → 要件定義 → 実装
```

各フェーズの詳細は `references/` ディレクトリを参照。

## 1. 現状分析フレームワーク（As-Is分析）

### 1.1 業務整理の4ドキュメント

レガシーシステムの現状を把握するために、以下の4資料を作成する：

1. **機能構成図** — 最上位文書。全資料はここに紐づけて体系化する
2. **業務一覧** — 担当者の作業レベルまで細分化
3. **業務フロー** — BPMN2.0 レベル1で作成（可読性優先）
4. **機能一覧** — 現行システムの機能を網羅的に記録

**重要原則**: 業務は「誰が・何をトリガーに・どのような作業をしているか」を
担当者の作業レベルまで細分化し、BPR（業務プロセス再構築）のインプットとする。

### 1.2 ステークホルダー分析

行政統計調査システムの典型的な利用者は3者：

| 利用者 | 人数規模 | 特性 |
|--------|----------|------|
| 行政職員（政策課等） | 約3名 | 業務知識は高いが人事異動あり |
| 運用事業者 | 約10名 | 業務熟練だがITリテラシーにばらつき |
| 報告事業者（調査対象） | 約250社 | ITリテラシーは各社で大きく異なる |

### 1.3 典型的なレガシー課題パターン

行政統計調査システムで頻出する課題類型：

**チャネル乱立問題**
- 作成手段: 専用プログラム、Excel/PDF、紙
- 提出手段: メール、e-Gov、FAX、郵送
- チャネルごとに異なる受領・取込処理が発生

**新旧システム併用問題**
- 旧システムからの機能移行が不完全で両方を使い続けている
- 帳票出力の大半が旧システムのマクロに依存
- エラーチェック・疑義照会も旧システムの画面を利用

**手動運用の蔓延**
- タスクスケジューラの手動起動
- マスタデータの手動二重登録
- 公表原稿のHTML手作業

## 2. 課題・施策の優先順位付けフレームワーク

### 2.1 難易度×重要度マトリクス

施策の優先順位を「難易度」と「重要度」の2軸で評価する。

**難易度の評価軸**（6観点）:
- 既存プロセスの変更/新規事業の導入
- 組織体制の整備
- システム構築
- インフラやデータの整備
- 制度・ルールの変更
- 職員のスキル開発/組織改革

**重要度の評価軸**（3観点）:
1. **継続性**: 既存の業務・機能を次期システムでも漏れなく実現できるか
2. **安定性**: 業務品質に大きな影響を与える課題を解消できるか
3. **効率性**: 非効率な業務を次期システムで改善できるか

**判定ルール**: 難易度「低〜中」かつ重要度「高」→ **短期的取組**（要件定義に反映）

## 3. 将来像設計（To-Be）の7本柱

次期システム更改で検討すべき主要テーマ：

### 3.1 調査票収集チャネルの統合
- **Web入力フォーム**を新規開発し、利用を推奨
- 段階的移行: メール・e-Gov → 廃止、郵送 → 残存、FAX/Excel添付 → 緊急措置のみ
- バリデーションチェック、未記入防止、企業情報の自動表示

### 3.2 業務効率化に向けた機能充足
- 20以上の不足機能を洗い出し実装（詳細は references/feature-gap.md）
- タスクスケジューラ → 画面操作への変更
- 画面操作性の3分類: データ編集画面、データ参照画面、機能実行画面

### 3.3 認証方法の設計
- デジタル庁「本人確認ガイドライン」に基づく保証レベル判定
- 特定法人への調査 → 保証レベル1（ID/パスワード認証）が適切
- 身元確認: 石油輸入登録簿等で法人確認済み → 追加確認不要

### 3.4 業界団体システムとの連携
- 業界団体の報告入力プログラムで作成したファイルをWebフォームでアップロード可能に
- 統計法の制約上、提出先の一元化は将来課題として段階的に対応

### 3.5 クラウド化実施方針
- クラウド・バイ・デフォルト原則に基づきクラウド利用を前提
- ブラウザ経由で行政側からもシステムに接続可能にする
- ISMAP対応クラウドサービスの選定

### 3.6 データ利活用促進
- 収集データの公開方法・形式の最適化
- API提供やダッシュボード化の検討
- エビデンスに基づく政策形成（EBPM）への対応

### 3.7 非機能要件
- 可用性、性能、セキュリティ、移行性等を標準ガイドライン群に準拠して定義

## 4. 実装ガイドライン

### 4.1 推奨技術スタック

```
フロントエンド: React (TypeScript) + TailwindCSS
バックエンド:   Python (FastAPI / Django)
データベース:   PostgreSQL
認証:           ID/パスワード（保証レベル1）、将来的にGビズID連携
インフラ:       AWS / Azure / GCP（ISMAP登録済みサービス）
CI/CD:          GitHub Actions
```

### 4.2 Web入力フォームの設計原則

```
1. インストール不要 — ブラウザのみで完結
2. 企業情報の自動表示 — ログイン時のアカウント情報から自動補完
3. バリデーション — 未記入・形式不正を提出前にチェック
4. 業界団体ファイル対応 — .sid/.GENファイルのアップロード受付
5. 修正・取下げ — 提出後の修正・取下げ機能と変更履歴
6. フォーマット編集 — 管理者が調査票フォーマットを動的に変更可能
7. オフライン配慮 — 郵送チャネルは当面残し段階的にオンライン移行
```

### 4.3 データ処理パイプライン設計

```
調査票受領 → バリデーション → 個票DB登録 → 集計処理 → 月報DB更新
    → エラーチェック → 疑義照会 → 修正 → 帳票生成 → 公開
```

重要: 各ステップを画面操作で実行可能にし、タスクスケジューラの手動起動を排除する。

### 4.4 移行戦略

段階的な移行アプローチを採用：

```
Phase 1 (Can-Be): Web入力フォーム新規開発 + 既存チャネル一部残存
Phase 2 (To-Be):  全チャネルをオンラインに統合、旧システム完全廃止
```
